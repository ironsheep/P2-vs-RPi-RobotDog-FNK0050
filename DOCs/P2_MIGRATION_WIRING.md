Goal: drop a **Propeller 2 Edge Module + experiment/breakout board** in where the
Raspberry Pi sits, via an adapter that mates to the connection board's 40-pin header.

**Hardware on this build: PCB v1.0** (so the WS2812 LED data line is on **GPIO18**, the
`rpi_ws281x` path — see [`RPI_GPIO_USAGE.md`](RPI_GPIO_USAGE.md)).

> ⚠️ **Confidence levels.** *Signal* assignments are **code-derived** (certain — traced in
> the server source). Power *direction* is now **confirmed by Freenove's docs** (§1). The
> exact *rail voltages* (5 V / 3.3 V) and the 3.3 V provenance are still **inferred** — no
> schematic exists — so those carry a "**verify**" step. Measure before you wire.

## 1. Direct answers

**Who powers whom?** The **robot powers the controller**, not the other way around —
**confirmed by Freenove's docs**: *"If you disconnect cable, the batteries will power
Raspberry Pi"* and *"all servos connect to the shield for power."* The battery pack feeds a
regulator on the connection board (shield), which delivers **5 V into the Pi's 40-pin
header** (pins 2 & 4). The Pi never powered the servos — 13 servos pull amps, far beyond
what a controller can source.

**Two power switches** (from the assembly docs): pressing **S1 and S2** lights two
indicators. **S1 = "Load"** gates the **servo** power rail (the docs have you toggle Load
off to install screws so the servos aren't energized); the other switch gates main/logic
(Pi) power. ➜ For P2 bring-up this is a gift: you can power the **P2 + logic with the Load
switch OFF** to test I²C, sensors, and code with the legs de-powered, then enable servos.

**USB caveat:** a USB cable can power the Pi *in parallel* with the battery (*"you can still
use cable to power Raspberry Pi with switches ON"*) — but **USB does not charge the
batteries** (see Battery & charging below).

**What voltage is on the connector?**
- **5 V** — sourced *by the robot* (battery → board buck regulator), present on header
  **pins 2 & 4**. This is your input power for the P2 board.
- **3.3 V** — on header **pins 1 & 17**, but historically **generated by the Pi** and
  handed *out* to the connection board's logic/I²C pull-ups. **Remove the Pi and this rail
  vanishes** — your P2 adapter must *supply* 3.3 V back onto these pins if the board's I²C
  chips/pull-ups depend on it (very likely). This is the #1 gotcha.
- Battery voltage (7.4 V nom / 8.4 V full / 6.4 V cutoff — from `ADS7830.py`,
  `Server.battery_reminder`) stays on the board's servo/sensor rails; it is **not** exposed
  on the 40-pin header.

**5 V or 3.3 V for the P2?** The P2 is a **3.3 V part and its I/O is NOT 5 V tolerant**
(pin abs-max ≈ 3.6 V). You feed the P2 board **5 V** (it regulates to 3.3 V on-board), and
every *signal* line into a P2 pin must be ≤ 3.3 V. The good news: the Pi was also 3.3 V, so
the board was already built for 3.3 V logic — but the two 5 V-native sensors (ultrasonic,
WS2812) are the ones to watch (§4).

## 1b. Battery & charging — getting the device ready for use

Sourced from `About_Battery.pdf` (in this repo) + the FNK0050 docs + the software's own
voltage thresholds.

- **Cells:** **2× 18650 Li-ion**, **unprotected**, flat-top *or* button-top. **Not
  included** — buy separately (Freenove battery list above). Wired **2-series (2S)**.
- **Voltage (from `ADS7830.py` / `Server.py`):** **8.4 V** full (2 × 4.2 V), **7.4 V**
  nominal, software **low-battery cutoff at 6.4 V** → at/below that the server relaxes the
  legs and exits. The ADC reads the pack through a ÷2 divider (`data/255 × 5.0 × 2`).
- **⚠️ No on-board charging.** Direct quote: *"The control board connected to the USB cable
  will **not** charge the batteries. So you also need a charger."* You **remove the 18650
  cells and charge them in an external 18650 charger** (*"almost any charger suitable for
  18650 batteries can be used"*), then reinstall. There is **no charge port, no charge IC,
  and no in-place charging** on this board.
- **Readiness checklist before powering the robot:**
  - [ ] Charge both 18650s **fully (~4.2 V each / 8.4 V pack)** in an external charger.
  - [ ] Observe **polarity** — *"Don't reverse batteries. Or the board will be damaged."*
  - [ ] Insert cells **before** running servos (*"prepare the right batteries and fully
        charge them before assembling … without right batteries will … damage the servos"*).
  - [ ] Use a matched pair (same brand/capacity/charge state) for a 2S pack.
- **Implication for the P2 port:** the power-readiness story is **unchanged** by swapping in
  the P2 — you still hand-charge 18650s externally and the battery still feeds the shield's
  5 V regulator. The P2 board simply replaces the Pi as the 5 V load. If you want true
  in-place / USB charging you'd have to **add** a 2S charge circuit (e.g. a BMS/charger
  module) yourself; the kit has none.

## 2. Power architecture (rails inferred — verify each arrow)

```
  2S Li-ion pack ──► Board buck reg ──► 5V rail ─┬─► header pins 2,4  ──► (was) Pi 5V in
   7.4–8.4V         (on conn. board)             ├─► servo V+ (PCA9685 power)
                                                 ├─► HC-SR04 Vcc (5V)
                                                 └─► WS2812 Vcc (5V)

  Pi onboard 3.3V reg ──► header pins 1,17 ──► board I²C pull-ups + PCA9685/ADS7830/MPU6050 Vcc(logic?)
       (DISAPPEARS when the Pi is removed — P2 board must replace this source)
```

**Verify with a multimeter (Pi removed, battery off unless noted):**
1. **5 V rail** — power the board, measure header pin 2/4 → GND. Expect ~5 V. Confirm the
   regulator is rated for the P2's draw (P2 ≈ 0.1–0.4 A; trivially less than the Pi it
   replaced, so almost certainly fine).
2. **3.3 V provenance** — continuity/resistance from the I²C chips' Vcc and the SDA/SCL
   pull-up resistors to header **pins 1/17** vs. to the **5 V** rail. This tells you whether
   those chips run on 3.3 V (you must re-supply it) or on 5 V (then their I²C is pulled to
   5 V → you need level shifting — see §4).
3. **I²C pull-up rail** — measure SDA (pin 3) and SCL (pin 5) idle voltage with the board
   powered and a known-good 3.3 V applied to pins 1/17. Must idle at ~3.3 V, not 5 V.
4. **HC-SR04 ECHO divider** — check for a resistor divider on ECHO (pin 15) between the
   sensor and the header. If raw 5 V reaches the header pin, **do not** connect it to the
   P2 without adding one (§4).

## 3. Wiring map: connection-board header → P2

Signal columns are **certain** (from code). The **P2-pin column is a committed
assignment**: every signal maps to **P2 pins P8–P15**, ordered to mirror the robot
connector's physical layout with **minimal wire crossover** — **P8 at the far end** of
the connector, climbing to the **VCC end where the I²C pair lands on P14/P15**. Rows
below run **far end → VCC end** to match the routing.

**Why this is low-crossover:** of the six signals, **five are on the header's odd row**
(ECHO 15, TRIG 13, Buzzer 11, SCL 5, SDA 3) and map straight across in order with no
crossings. Only the **WS2812 LED (pin 12) is on the even row** — it's the single
cross-row jump. The unused **P12–P13** fall in the header's own physical gap (pins
7–10), so the spacing stays faithful end-to-end.

| Conn-board header (far→VCC) | BCM signal | Function (certain) | P2 pin | Notes |
|---|---|---|---|---|
| Pin 15 *(far end)* | GPIO22 | **Ultrasonic ECHO** (input) | **P8** | ⚠️ 5 V sensor — **must arrive ≤ 3.3 V** (§4) |
| Pin 13 | GPIO27 | **Ultrasonic TRIG** | **P9** | P2 output (3.3 V triggers HC-SR04 fine) |
| Pin 11 | GPIO17 | **Buzzer** drive | **P10** | P2 output; safe |
| Pin 12 | GPIO18 | **WS2812 LED data** (PCB v1.0) | **P11** | only **even-row** signal → the one expected cross-row jump; 3.3 V data into 5 V strip (§4) |
| — | — | *(spare — header's physical gap, pins 7–10)* | **P12–P13** | unassigned; available for future I/O |
| Pin 5 (SCL1) | GPIO3 | **I²C SCL** (same bus) | **P14** | 3.3 V open-drain; shares pull-ups |
| Pin 3 (SDA1) *(VCC end)* | GPIO2 | **I²C SDA** → PCA9685 (servos), ADS7830 (battery), MPU6050 (IMU) | **P15** | needs pull-ups (likely already on board) |
| Pins 2, 4 | 5V | **Power IN from robot** | → P2 board **5 V input** | sized for the regulator (verify) |
| Pins 1, 17 | 3.3V | **3.3 V the board's logic needs** | ← from P2 board **3.3 V out** | feed *back* to board (verify §2.2) |
| Pins 6,9,14,20,25,30,34,39 | GND | Ground | **common GND** | tie robot GND ↔ P2 GND |

> The Buzzer↔LED order on **P10/P11** is the one degree of freedom left (both sit at the
> same longitudinal header position, pins 11 & 12 directly across from each other). Swap
> them if the adapter's even-row routing prefers it.

**P2 Edge reserved pins — don't use for robot I/O:** the Edge module uses **P58–P61** for
the boot SPI flash and **P62/P63** for the serial/programming host. The **P8–P15** block
chosen above sits safely inside the usable **P0–P57** range. The experiment/breakout board
supplies the module's 3.3 V from a 5 V input — that's the regulator you feed from the
robot's 5 V and tap to re-supply header pins 1/17.

## 4. 3.3 V vs 5 V hazards (P2 is NOT 5 V tolerant)

| Line | Direction | Risk | Action |
|---|---|---|---|
| **I²C SDA/SCL** | bidir | If pull-ups go to **5 V**, the bus idles at 5 V → kills P2 pins | Confirm pull-ups tie to 3.3 V (Pi-style). If 5 V, use a proper I²C level shifter (e.g. BSS138 bidirectional) |
| **Ultrasonic ECHO** | sensor→P2 | HC-SR04 echo is a **5 V** output | Confirm/add a divider so the P2 sees ≤ 3.3 V (e.g. 1 kΩ series + 2 kΩ to GND, or a level shifter). Or run the sensor at 3.3 V if it's a 3.3 V-tolerant variant |
| **Ultrasonic TRIG** | P2→sensor | P2 drives 3.3 V; HC-SR04 trigger threshold usually OK | Safe; if flaky, level-shift up to 5 V |
| **Buzzer** | P2→board | Driving a transistor/driver from 3.3 V | Safe (output) |
| **WS2812 data** | P2→strip | 3.3 V data into a 5 V strip (data-high threshold ≈ 0.7·Vcc) | The Pi did this at 3.3 V already; usually works. If the first pixel is unreliable, add a 3.3→5 V level shifter (74AHCT125) or power the strip at ~4.3 V |

P2 pins have **configurable drive strength** (1.5 kΩ / 15 kΩ / ~1 mA / actual-drive
modes); set adequate drive for TRIG, buzzer, and the LED data line in firmware.

## 5. What stays the same on the firmware side

Most of the robot is on **I²C**, so porting is mostly "implement an I²C master on the P2":
- **Servos** (PCA9685 @ `0x40`), **battery ADC** (ADS7830 @ `0x48`), **IMU** (MPU6050 @
  `0x68`) — all spoken to over the *same* I²C bus the Pi used. The P2 just needs an I²C
  master (smart-pin or bit-banged; OBEX has I²C objects). Register sequences, the 50 Hz
  servo timing, and the channel→leg map are documented in `CLAUDE.md` / `RPI_GPIO_USAGE.md`
  and translate 1:1.
- Only the **3 raw-GPIO devices** (buzzer, ultrasonic, WS2812) need P2-native pin code —
  the buzzer is trivial, the ultrasonic is a TRIG pulse + ECHO timing (a natural fit for a
  P2 smart pin in pulse-measure mode), and the WS2812 is a classic P2 smart-pin/streamer task.

## 6. Pre-wiring checklist

- [ ] Meter the **5 V** rail on header pins 2/4; confirm the board regulator's current headroom.
- [ ] Determine **3.3 V provenance** (pins 1/17): is board logic powered from it? If yes,
      plan to source 3.3 V from the P2 board back onto those pins.
- [ ] Confirm **I²C pull-up rail** is 3.3 V (not 5 V).
- [ ] Confirm/add the **ECHO divider** so the P2 never sees 5 V.
- [ ] Establish a single **common ground**.
- [ ] Keep robot signals on **P0–P57**; leave **P58–P63** for flash/serial.
